# Product Vision: Meeting-To-Diff Agent Sessions

## Vision

Get work done by talking about it in a meeting.

The app should feel like a text-first work meeting where the conversation is the primary interface,
the agent is an active participant, and the outcome is completed work in a repository workspace.

The core product loop is:

1. People join a shared session for a repository.
2. They discuss the work in a meeting transcript.
3. The app turns meeting intent into suggested agent prompts.
4. The controller approves the agent action.
5. The agent works in the session workspace.
6. The meeting ends with a reviewable artifact: transcript, agent output, changed files, and diff.

## Product Principles

- Conversation is the source of product intent.
- The transcript is not side chat; it is the durable record of what the team asked for.
- Agent action should be visible, attributable, and approved by the controller.
- The app should prefer suggested prompts over hidden autonomy for the MVP.
- The artifact matters more than the meeting notes: the session should produce a reviewable diff.
- HTTP should expose current state; WebSocket should stream live updates.
- The frontend should initialize from snapshots, then apply typed events incrementally.

## MVP Scope

The MVP is a text-only implementation meeting that produces a reviewable diff.

In scope:

- Create and start a session for a Git repository.
- Join as a viewer.
- Claim controller role.
- Post meeting chat messages.
- Generate deterministic prompt suggestions from chat.
- Accept or dismiss prompt suggestions as the controller.
- Send accepted suggestions to the agent runtime.
- Stream agent status and output live.
- Show an active working state while the agent is running before output arrives.
- Show the session artifact summary.
- Show changed files and raw git diff.
- Refresh workspace review when agent work finishes or files change.

Out of scope for the MVP:

- Pull request creation.
- Commit creation.
- Full meeting lifecycle states such as scheduled, active, ended, archived.
- Real LLM-powered prompt suggestion generation.
- Audio/video meetings.
- Multi-agent orchestration.
- Rich decision/open-question extraction.

## Current MVP Behavior

Prompt suggestions are generated by a deterministic fake generator. When a joined viewer posts a
chat message containing implementation intent words such as `add`, `build`, `implement`, `fix`, or
`write`, the app creates a pending prompt suggestion if there is no existing pending suggestion.

The current demo suggestion is:

```text
Add hello world to the README.
```

Only the controller can accept or dismiss a prompt suggestion. Accepting it marks the suggestion as
accepted and sends the suggestion text through the same agent prompt path as a manual prompt.

The review surface shows:

- chat message count
- changed file count
- suggestion count
- artifact status
- changed files
- raw diff

## Next Product Gaps

- Replace deterministic prompt suggestions with a non-interactive LLM adapter.
- Feed richer context into prompt generation: transcript, agent state, recent agent output, changed
  files, and diff summary.
- Make agent replies feel more like part of the meeting transcript.
- Add an explicit end-meeting/finalize action that creates a durable artifact summary.
- Add clearer stuck/slow/error states for the agent runtime.
- Add provenance from chat messages and accepted suggestions to workspace changes.
